# Managed Agent v0.2 技术方案

## 0. 核心原则

- **Claude Agent SDK 保留**，用它驱动 LLM wiki 的 ingest/query
- **LLM wiki 作为"加工层"**：narrative 内容在 `wiki/` 的 markdown 文件，按 Karpathy skill 的规范维护
- **SQLite 作为"结果层"**：结构化字段（stage、contacts、record 元信息）从 wiki 提取后落地，支撑 H5 列表/筛选/查询
- **raw/ 作为"档案层"**：docx 原文永久留底，不修改
- **Git 作为版本控制**：wiki/ 和 raw/ 都在 git 里，history 白捡

三层职责：

```
raw/      不可变档案      （docx 原文落盘）
 ↓
wiki/     narrative 演化  （LLM 合并、重写、级联）
 ↓
SQLite    结构化索引      （从 wiki 提炼的字段 + 关系）
```

## 2. 数据模型（SQLite）

### 2.1 PRAGMA 设置（连接时执行）

```python
await conn.execute("PRAGMA journal_mode=WAL")          # 并发关键：读不阻塞写
await conn.execute("PRAGMA synchronous=NORMAL")        # WAL 下够安全
await conn.execute("PRAGMA busy_timeout=5000")         # 遇锁等 5 秒
await conn.execute("PRAGMA foreign_keys=ON")           # 外键启用
await conn.execute("PRAGMA temp_store=MEMORY")
await conn.execute("PRAGMA mmap_size=268435456")       # 256MB mmap
await conn.execute("PRAGMA cache_size=-65536")         # 64MB page cache
```

### 2.2 MVCC 说明

SQLite WAL 模式 ≈ 快照隔离的 MVCC：

- **读事务拿一个 snapshot**，期间看到的是事务开始那一刻的数据。即使别人在写，读者也不会被阻塞，也看不到未提交的变化
- **写事务最多 1 个同时进行**（SQLite 是单写多读）
- 写者不阻塞读者，读者不阻塞写者
- 并发写 → `SQLITE_BUSY` → `busy_timeout` 自动重试

**我们的并发策略：**

| 场景 | 策略 |
|------|------|
| 多 H5 请求读客户列表 | 并发无锁，每个请求一个 snapshot |
| 同一客户的两次 ingest | in-process asyncio.Lock 串行化（避免 wiki 文件互相覆盖） |
| 不同客户的并发 ingest | 并行执行，但全局并发上限（semaphore，限 3-5 路），防 Anthropic rate limit |
| SQL 写 | 短事务 + `BEGIN IMMEDIATE`，配合 busy_timeout 自动重试 |

不用上 PostgreSQL / Redis，SQLite WAL 够用到几千并发读 + 几十并发写。

### 2.3 Schema

```sql
-- ============ 客户 ============
CREATE TABLE customers (
    id            TEXT PRIMARY KEY,              -- slug: "upe", "bytedance"
    name          TEXT NOT NULL UNIQUE,          -- 显示名: "UPE"
    aliases       TEXT NOT NULL DEFAULT '[]',    -- JSON array
    industry      TEXT,
    size          TEXT,
    summary       TEXT NOT NULL DEFAULT '',      -- 从 wiki 提取的短摘要(300-800字)
    journey_stage TEXT NOT NULL,
    wiki_path     TEXT NOT NULL,                 -- "wiki/customers/upe.md"
    created_at    TEXT NOT NULL,
    created_by    TEXT,                           -- open_id
    updated_at    TEXT NOT NULL,
    CHECK (journey_stage IN (
        '线索','初步接触','需求确认','方案沟通',
        '报价商务','合同签约','交付验收','运营维护','流失冻结'
    ))
);
CREATE INDEX idx_customers_stage   ON customers(journey_stage);
CREATE INDEX idx_customers_updated ON customers(updated_at DESC);

-- ============ 联系人 ============
CREATE TABLE contacts (
    id           INTEGER PRIMARY KEY AUTOINCREMENT,
    customer_id  TEXT NOT NULL REFERENCES customers(id) ON DELETE CASCADE,
    name         TEXT NOT NULL,
    title        TEXT,
    phone        TEXT,
    email        TEXT,
    wechat       TEXT,
    notes        TEXT,
    created_at   TEXT NOT NULL,
    updated_at   TEXT NOT NULL,
    UNIQUE(customer_id, name)
);
CREATE INDEX idx_contacts_customer ON contacts(customer_id);

-- ============ 跟进记录 ============
CREATE TABLE followup_records (
    id                   INTEGER PRIMARY KEY AUTOINCREMENT,
    customer_id          TEXT NOT NULL REFERENCES customers(id) ON DELETE CASCADE,
    source_type          TEXT NOT NULL,          -- 'docx' | 'manual'
    source_url           TEXT,
    source_doc_id        TEXT UNIQUE,            -- 飞书 doc_id，去重
    source_title         TEXT,
    meeting_date         TEXT,                   -- ISO8601
    raw_path             TEXT NOT NULL,          -- "raw/customers/2026-04-21-xxx.md"
    summary              TEXT NOT NULL,          -- 2-4 句摘要
    journey_stage_after  TEXT NOT NULL,          -- 这次之后的阶段快照
    ingested_by          TEXT,                    -- open_id
    ingested_via         TEXT NOT NULL,          -- 'group' | 'h5'
    ingest_job_id        TEXT,
    created_at           TEXT NOT NULL
);
CREATE INDEX idx_records_customer_time
    ON followup_records(customer_id, meeting_date DESC, created_at DESC);

-- ============ 客户 ↔ 飞书群 ============
CREATE TABLE customer_lark_groups (
    chat_id       TEXT PRIMARY KEY,
    customer_id   TEXT NOT NULL REFERENCES customers(id) ON DELETE CASCADE,
    bound_at      TEXT NOT NULL,
    bound_by      TEXT NOT NULL                  -- open_id
);
CREATE INDEX idx_groups_customer ON customer_lark_groups(customer_id);

-- ============ 飞书用户（登录态） ============
CREATE TABLE users (
    open_id       TEXT PRIMARY KEY,
    union_id      TEXT,
    name          TEXT,
    avatar_url    TEXT,
    last_login_at TEXT,
    created_at    TEXT NOT NULL
);

-- ============ Web 会话 ============
CREATE TABLE sessions (
    token         TEXT PRIMARY KEY,               -- 随机 32 字节 hex
    open_id       TEXT NOT NULL REFERENCES users(open_id) ON DELETE CASCADE,
    expires_at    TEXT NOT NULL,
    created_at    TEXT NOT NULL
);
CREATE INDEX idx_sessions_open_id ON sessions(open_id);

-- ============ Ingest 任务（异步） ============
CREATE TABLE ingest_jobs (
    id                 TEXT PRIMARY KEY,          -- uuid4
    customer_id        TEXT REFERENCES customers(id),
    triggered_via      TEXT NOT NULL,             -- 'group' | 'h5'
    triggered_by       TEXT NOT NULL,             -- open_id
    source_url         TEXT NOT NULL,
    source_doc_id      TEXT,
    status             TEXT NOT NULL,             -- queued|fetching|ingesting|committing|done|failed
    error              TEXT,
    result_record_id   INTEGER REFERENCES followup_records(id),
    feishu_card_id     TEXT,                       -- 用于异步更新卡片
    feishu_chat_id     TEXT,
    created_at         TEXT NOT NULL,
    updated_at         TEXT NOT NULL,
    finished_at        TEXT
);
CREATE INDEX idx_jobs_status     ON ingest_jobs(status, created_at);
CREATE INDEX idx_jobs_customer   ON ingest_jobs(customer_id, created_at DESC);
```

**关键字段解释：**

- `customers.wiki_path`：LLM wiki 文章路径。SQLite 行 ↔ wiki 文章是 1:1 映射
- `customers.summary`：**不是 wiki 全文**，是从 wiki 提炼的 300-800 字摘要，给 H5 列表 hover 和群里快速回答用
- `followup_records.raw_path`：docx 原文落在 raw/ 的路径，点击可追溯
- `followup_records.journey_stage_after`：每条记录生效后的 stage 快照，这样能画一条"旅程推进"时间线
- `ingest_jobs`：异步队列表，H5 提交后立即返回 job_id，前端轮询；群里的"处理中..."卡片也通过它异步更新

## 3. LLM Wiki 使用方式

### 3.1 映射

```
1 customer = 1 wiki article
customers.id = wiki 文章 slug = wiki/customers/<slug>.md
```

### 3.2 Ingest Pipeline 完整流程

```
Trigger（三条路）:
  A. 群 @bot 贴 URL       →  chat_id 已绑定 → customer_id
  B. H5 客户详情页"加记录" →  customer_id from URL
  C. H5 客户列表页"加记录" →  弹窗先选客户 → customer_id
  （新建客户是独立操作，与 ingest 解耦）
                ↓
[1] 创建 ingest_job (status=queued) → 立即返回 job_id
    - 群里回卡片"📥 排队中..."（存 card_id 到 job）
    - H5 返回 job_id → 前端 HTMX 轮询 /jobs/{id}
                ↓
[2] 后台 worker 拿 job：
    2a. 获取 per-customer asyncio.Lock
    2b. lark-cli 抓 docx (status=fetching)
    2c. 去重检查：source_doc_id 已存在 → 直接 done + 提示
                ↓
[3] Ingest 阶段 (status=ingesting)：
    调 Claude Agent SDK：
      options = ClaudeAgentOptions(
          system_prompt=INGEST_PROMPT,         # 引用 LLM wiki skill
          model="claude-haiku-4-5",
          allowed_tools=["Read","Write","Edit","Glob","Grep"],
          cwd=PROJECT_ROOT,                     # wiki/ 和 raw/ 的根
          permission_mode="bypassPermissions"   # 受控环境
      )
      user_msg = f"""
      Ingest 这份飞书会议纪要到 LLM wiki。
      客户：{customer.name} (id={customer.id}, 已有文章 {wiki_path})
      会议日期：{meeting_date}
      来源 URL：{source_url}

      按 karpathy-llm-wiki skill 的 ingest 流程：
      1. 先把原文落到 raw/customers/{YYYY-MM-DD}-{slug}.md
      2. 读 wiki/customers/{customer.id}.md，基于它和新材料合并更新
      3. 检查 index.md 和 log.md，更新
      4. 不要级联到其他客户文章（customer 间不交叉）

      原文：
      {docx_raw_content}
      """
      # agent 自主用 Read/Write/Edit 完成，我们消费流返回值即可
                ↓
[4] Extract 阶段 (status=committing)：
    单次 structured output（不走 agent loop）：
      input = 更新后的 wiki/customers/<id>.md 全文 + 本次 raw 文件
      schema:
        {
          "summary": str (300-800字),
          "journey_stage": enum,
          "record_summary": str (2-4句),
          "contacts_delta": [
            {"action": "add|update", "name": str, "title"?: str,
             "phone"?: str, "email"?: str, "notes"?: str}
          ]
        }
    便宜：单次 call，~10K input，2K output。
                ↓
[5] 事务写 SQL:
    BEGIN IMMEDIATE;
      INSERT followup_records (..., raw_path, summary, journey_stage_after);
      UPDATE customers SET summary=?, journey_stage=?, updated_at=?;
      UPSERT contacts per delta;
      UPDATE ingest_jobs SET status='done', result_record_id=?, finished_at=?;
    COMMIT;
                ↓
[6] Git commit wiki/ 和 raw/（subprocess）：
    git add raw/ wiki/
    git commit -m "ingest: {customer.id} / {record_summary_first_line}"
                ↓
[7] 释放 lock，通知：
    - 群里更新卡片为"✅ UPE 已更新：需求确认 → 方案沟通"
    - H5 轮询端返回 done + 跳新记录
```

### 3.3 Query 流程

```
群里 @bot 问 "UPE 整体进展怎么样"
      ↓
[1] 解析：chat_id → customer_id
[2] 拼 context (src/query/context.py):
    SQL: customer 基本信息 + 最近 10 条 records + contacts
    File: 读 wiki/customers/<id>.md 全文
[3] Claude Agent SDK query (可带工具，让它能翻更多 raw/):
    model = "claude-sonnet-4-6"  # query 稍微聪明点
    system: "你是客户情报助手..."
    allowed_tools = ["Read", "Glob", "Grep"]
    prompt: context 注入 + 用户问题
[4] 流式回卡片
```

Query 用 Agent SDK 的价值：用户可能问很深的问题（"他们技术栈是啥"、"上次吐槽啥"），agent 可以自己去 raw/ 里翻原始会议纪要找答案，而不是只看合成后的 wiki。

## 4. 三类锁 / 并发 / 错误处理

### 4.1 锁层级

```python
# src/ingest/lock.py
class CustomerLocks:
    _locks: dict[str, asyncio.Lock] = {}
    _global = asyncio.Semaphore(3)  # 全局并发 ingest 上限

    @classmethod
    @asynccontextmanager
    async def acquire(cls, customer_id: str):
        async with cls._global:                 # 限全局 rate
            lock = cls._locks.setdefault(customer_id, asyncio.Lock())
            async with lock:                    # 限同客户串行
                yield
```

**排队行为：**

```
同一客户 A：
  Req1 进来 → 拿 lock，开跑
  Req2 进来 → 在 lock 上排队等
  Req3 进来 → 继续排在 Req2 后面
  Req1 跑完释放 → Req2 开始
  ...

不同客户 A / B / C：
  并行跑，但总并发 ≤ Semaphore(3)
  第 4 个不同客户的 req 要等前面某个跑完
```

**同客户必须串行的原因：**

- 两个 req 同时改 `wiki/customers/upe.md` 会互相覆盖（后写赢，前面合并的新信息丢）
- 两个 req 同时 `UPDATE customers SET summary=...` 也会互相覆盖
- raw/ 下同客户同天两份文件会撞名

**用户侧体验：**

- 群里：卡片显示 "📥 排队中 (前面还有 N 个)..." → "处理中..." → "✅ 已更新"
- H5：前端 HTMX 轮询 `/jobs/{id}` 实时更新状态

**全局 Semaphore(3) 原因：** Anthropic rate limit，同时 5 个 ingest 跑会挤爆 tokens/min。3 是保守值，后期升 tier 再调。

### 4.2 事务模式

```python
async with db.transaction():  # 内部 BEGIN IMMEDIATE
    await records.insert(...)
    await customers.update(...)
    await contacts.upsert_many(...)
    await jobs.mark_done(...)
# 自动 COMMIT 或 ROLLBACK
```

### 4.3 失败恢复

- ingest_jobs 有 status，worker 启动时扫 `status IN (fetching,ingesting,committing)` 的 stale 任务（>10 分钟没更新）→ 重新入队或标 failed
- wiki/ 和 raw/ 由 git 兜底，坏了能 checkout
- SQL 和 git 不同步风险：**先 SQL commit 再 git commit**。如果 git commit 失败，下次 ingest 前检查工作区有未提交的变更会自动补 commit

## 5. H5 页面设计

### 5.1 设计原则

- **手机优先**：飞书小程序容器内是 H5，默认窄列单列
- **极简**：不做筛选、排序、新建客户等复杂控件；靠搜索 + 默认排序解决 90% 需求
- **现代视觉**：无边框、轻阴影、大圆角、中间点分隔、字段两端对齐
- **SSR + HTMX**：Jinja2 服务端渲染，HTMX 做局部刷新，避免 SPA 状态管理
- **无 SPA 路由**：每个页面就是一条真实 URL，便于飞书分享/深链
- **客户数据只读**：客户档案从外部 CRM 同步，H5 不新建、不删除客户

### 5.2 路由总览

**页面路由（返回 HTML）：**

```
GET  /                               → 未登录 → /login；已登录 → /customers
GET  /login                          → 登录页（空壳 + JSSDK JS）
GET  /customers                      → 客户列表
GET  /customers/{id}                 → 客户详情
GET  /customers/{id}/wiki            → 完整 wiki 渲染
GET  /customers/{id}/wiki/history    → wiki 版本历史（git log）
GET  /customers/{id}/wiki/diff?c=xx  → 版本 diff
```

**API 端点（返回 JSON 或 HTMX 片段）：**

```
POST   /api/auth/exchange            → auth_code → session cookie
PATCH  /customers/{id}               → 编辑基本信息 / stage
POST   /customers/{id}/records       → 提交 URL → 创建 ingest_job
GET    /jobs/{id}                    → 轮询 job 状态（HTMX 片段）
POST   /customers/{id}/contacts      → 加联系人
PATCH  /contacts/{id}                → 改联系人
DELETE /contacts/{id}                → 删联系人
POST   /customers/{id}/lark-group    → 建飞书群

POST   /internal/crm-sync            → CRM 同步入口（webhook 或定时任务触发）
```

（无 `POST /customers`：客户由 CRM 同步，H5 不创建。）

**HTMX 片段端点：**

```
GET /_p/customer-list          → 列表卡片（支持 ?q=&tab=）
GET /_p/job-progress/{id}      → job 进度条
GET /_p/record-card/{id}       → 单条记录展开/收起
```

### 5.9 错误 / 空状态

- **客户列表为空**（CRM 未同步）：提示"暂无客户，请确认 CRM 已同步数据"（不提供"新建客户"入口）
- 搜索无结果：提示"未找到 {q}，请确认名称或去 CRM 查询"
- 详情页无跟进：提示"这个客户还没有跟进记录" + [+ 加第一条记录]
- Ingest 失败：红字显示 error + [重试]
- 群里贴 URL 但群未绑定：bot 回"此群未绑定客户，请先 `@bot /bind <客户名>`"
- Session 过期：页面顶部横幅"登录已过期" → 自动刷新走 login 流程

## 6. 成本

Haiku 4.5 估算：

- Ingest agent loop (~4 轮): ~50K input + 5K output ≈ **$0.08/doc**
- Extract 单次 structured: ~10K input + 2K output ≈ **$0.02/doc**
- Query: 依 context 大小 ~$0.01-0.03/次

总 ingest 成本 **~$0.10/篇**。$500 够 5000 篇。

## 7. 阶段拆分

**Phase 1 — 数据层 + CRM 同步 + ingest 核心（基础）**

- SQLite 建表 + migration（customers 带 `crm_*` 字段，参考 `crm-integration-design.md` §2.2/2.3.2）
- aiosqlite 封装 + PRAGMA
- **CRM 同步**（`src/crm/`）：
  - `bytehouse.py` 只读 CH 客户端
  - `sync_valuelist.py` / `sync_users.py` / `sync_account.py` / `sync_stages.py`
  - 首次全量：70K accounts 从 `ODS_YL.crm_account` 镜像到本地
  - 15 分钟增量：`updatedAt > watermark`
  - APScheduler 调度
  - 夜间对账
- `src/ingest/pipeline.py` 全流程跑通
- `src/ingest/wiki_agent.py` 调 Claude Agent SDK + 装载 LLM wiki skill
- `src/ingest/extract.py` 结构化输出
- git auto-commit
- 验收：
  1. 初始化同步跑完，本地 `customers` 行数 ≈ CRM 远端 `is_crm_deleted=0` 行数
  2. 跑 15 分钟增量，增量行数 < 全量
  3. CLI 手动喂一个 docx URL + 已有 customer_id → 走完 ingest 全流程

**Phase 2 — 群 Bot 集成（接真实入口）**

- 群消息 @mention 解析
- `/bind`, `/unbind`, `/status` 命令
- URL ingest → 异步 job → 卡片更新
- Query 模式
- 砍掉 p2p

**Phase 3 — H5 前端（销售端 UI）**

- FastAPI + 飞书小程序 OAuth
- 客户列表 / 详情 / 加记录
- Job 轮询
- 建群按钮（`im.chat.create`）

**Phase 4 — 打磨**

- 联系人 CRUD
- Wiki 版本查看（调 git log）
- 失败恢复 UI
- 空状态、错误提示

## 8. 技术栈总结

| 层 | 技术 |
|----|------|
| LLM | Anthropic Haiku 4.5（ingest） / Sonnet 4.6（query） |
| Agent 框架 | Claude Agent SDK (`claude_agent_sdk`) |
| Wiki 规范 | Karpathy LLM Wiki skill (`.agents/skills/karpathy-llm-wiki/`) |
| DB | SQLite + WAL + aiosqlite |

| 前端容器 | 飞书小程序（H5 shell + JSSDK） |
| 认证 | 飞书小程序 auth code → session cookie |
| Bot | 飞书长连接（现有 `lark_client.py` 保留） |
| Docx 读 | `lark-cli --as user`（现有） |
| 版本控制 | git（wiki/ + raw/） |
| 进程 | 单 Python 进程同时跑 bot 长连接 + FastAPI + ingest worker + CRM sync 调度 |
| CRM 数仓 | ByteHouse (ClickHouse 兼容)，`clickhouse-driver` TLS 连接 |
| 同步调度 | APScheduler（进程内 asyncio） |
